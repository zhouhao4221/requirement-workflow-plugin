# 需求分析助手

仅在创建或编辑需求时激活，引导完善需求文档各章节。

## 分析框架

创建阶段（`/req:new`）填充需求定义章节（一~六），不填实现方案（十）。

### 1. 一、需求描述
- 业务目标：解决什么问题？
- 用户价值：给谁用？
- 客户场景：原始业务诉求
- 使用场景：何时使用？

### 2. 二、功能清单
- 拆分为原子功能点
- 每个功能点可独立开发/测试
- 标注依赖关系

### 3. 三、业务规则
| 类型 | 内容 |
|------|------|
| 数据校验 | 必填、格式、长度 |
| 状态转换 | 流转规则 |
| 权限控制 | 操作权限 |
| 边界条件 | 特殊情况 |

### 4. 四、使用场景
- 角色、前置条件
- 基本流程（用户操作 → 系统响应）
- 异常流程

### 5. 五、API 设计
- RESTful 风格
- 仅描述业务语义（接口名称、方法、路径、说明）
- 不涉及实现细节

### 6. 六、测试要点
- 正常流程
- 边界条件
- 异常情况
- 权限测试

### 不在创建阶段填写

- **十、实现方案**（含数据模型、文件改动清单、实现步骤）→ 在 `/req:dev` 阶段由 AI 分析代码后自动生成

## 输出要求

### 格式约束（强制）

生成或补充内容时，**必须严格遵循模板格式**：

1. **先读取模板**：根据需求类型读取对应模板文件
   - 正式需求：优先 `docs/requirements/template.md`，其次 `<plugin-path>/templates/requirement-template.md`
   - 快速修复：优先 `docs/requirements/quick-template.md`，其次 `<plugin-path>/templates/quick-template.md`
2. **章节结构不可变**：不得新增、删除、合并或重命名模板中的章节
3. **层级标题不可变**：章节标题、编号必须与模板完全一致
4. **表格格式不可变**：表格的列名、列数必须与模板一致
5. **保留空章节**：未涉及的章节保留模板占位文本，不得删除
6. **仅填充内容**：在模板对应章节的占位文本处填充实际内容

内容完整可执行。
