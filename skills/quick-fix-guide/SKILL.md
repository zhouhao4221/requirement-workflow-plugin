# 快速修复助手

快速分析问题，生成简洁可执行的实现方案。

## 触发场景

- 执行 `/req new-quick` 命令
- 用户说"这个bug"、"小问题"、"快速修一下"
- 用户说"简单加个功能"、"小改动"、"优化一下"
- 讨论简单的代码修复

## 分析流程

### 1. 快速理解问题

首先明确：
- **问题是什么**：错误信息、异常现象、功能缺失
- **现象 vs 根因**：表面现象背后的真实原因
- **影响范围**：涉及哪些功能、哪些用户

### 2. 快速定位代码

```python
# 定位策略（按优先级）
1. 根据报错信息/堆栈直接定位
2. 根据功能模块/路由定位
3. 搜索关键字/函数名/变量名
4. 根据日志信息定位
```

### 3. 分析现有实现

阅读相关代码，理解：
- 当前的实现逻辑
- 问题出在哪个环节
- 如何最小化改动修复

### 4. 生成实现方案

**输出格式：**

```
🔍 问题分析
[1-2句话说明问题根因]

💡 实现方案
[具体改动说明]

1. 文件：path/to/file.go
   改动：[具体改动内容]

2. 文件：path/to/file2.go
   改动：[具体改动内容]

📏 改动量：小/中（约 XX 行）

✅ 验证方式
[如何验证修复成功]
```

### 5. 方案评估

在生成方案后，自动检查：

| 检查项 | 标准 | 超出建议 |
|-------|------|---------|
| 文件数量 | ≤5 个 | 升级为正式需求 |
| 代码行数 | ≤100 行 | 考虑拆分 |
| 数据库变更 | 无 | 升级为正式需求 |
| 新增API | 无 | 升级为正式需求 |
| 影响范围 | 单一功能 | 评估风险 |

如果超出快速修复范围，主动提醒用户。

## 开发引导

### 代码规范（与 dev-guide 一致）

#### 文件命名
- 文件名：kebab-case（如 `sys-dept-channel.go`）

#### 日志规范
```go
// ✅ 正确
log.Info("操作成功", ctx, "id", id)
log.Error("操作失败", ctx, "error", err)

// ❌ 错误
fmt.Println("debug")
```

#### 错误处理
```go
// ✅ 正确
return errno.ErrDatabase
return errno.New(errno.ErrBadRequest, "自定义消息")

// ❌ 错误
return errors.New("xxx")
```

### 分层架构

即使是小改动，也要遵循分层架构：

```
Router → Controller → Biz → Store → Model
```

修改时确保：
- 不跨层调用
- 不在 Controller 写业务逻辑
- 不在 Biz 写 SQL

## 完成标准

### 基本要求
- [ ] 问题已解决/功能已实现
- [ ] 代码符合项目规范
- [ ] 简单自测通过

### 自测验证
- 复现原问题，确认已修复
- 测试正常流程，确认无副作用
- 测试边界情况（如适用）

## 常见场景模板

### Bug 修复

```
🔍 问题分析
[错误原因：空指针/逻辑错误/类型错误/...]

💡 修复方案
1. 文件：xxx.go:123
   改动：添加空值检查 / 修正判断条件 / ...

✅ 验证方式
1. 复现原问题场景
2. 确认错误不再出现
3. 确认正常流程不受影响
```

### 小功能添加

```
🔍 需求分析
[功能描述：在xxx处添加xxx能力]

💡 实现方案
1. 文件：xxx.go
   改动：新增 xxx 方法/字段

2. 文件：xxx.go
   改动：调用新方法

✅ 验证方式
1. 测试新功能正常工作
2. 测试不影响现有功能
```

### 代码优化

```
🔍 优化目标
[性能问题/代码重复/可读性/...]

💡 优化方案
1. 文件：xxx.go
   改动：[具体优化内容]

✅ 验证方式
1. 功能行为保持不变

```

## 交互方式

1. 快速理解用户描述的问题
2. 立即定位相关代码
3. 生成简洁的实现方案
4. 确认后直接开始开发
5. 完成后简单验证

**原则：快速、简洁、聚焦**
