# 需求测试

针对指定需求执行综合测试，包含三个层次：运行已有测试、引导创建新测试、交互验证测试要点。

> 存储路径和缓存同步规则见 [_common.md](./_common.md)

## 命令格式

```
/req:test [REQ-XXX] [选项]
```

- 省略编号时自动选择「开发中」或「测试中」的需求
- 多个候选时让用户选择

### 选项

| 选项 | 说明 |
|-----|------|
| `--failed` | 仅运行上次失败的测试 |
| `--skip-ut` | 跳过 UT 阶段 |
| `--skip-api` | 跳过 API 测试阶段 |
| `--skip-e2e` | 跳过 E2E 测试阶段 |
| `--force` | 某阶段失败时继续后续阶段 |

---

## 总体流程

```
1. 选择需求 & 前置检查
2. 提取测试要点（业务维度）
3. 识别变更范围（代码维度）
4. ━━ 阶段一：UT ━━
   ├── 回归：运行变更相关的已有 UT
   └── 新建：缺失时引导创建
5. ━━ 阶段二：API 测试 ━━
   ├── 回归：运行已有 API 测试
   └── 新建：缺失时引导创建
6. ━━ 阶段三：E2E 测试 ━━
   ├── 回归：运行已有 E2E 测试
   └── 新建：缺失时引导创建
7. 交互验证：自动化未覆盖的测试要点
8. 更新状态为「测试中」& 记录结果
9. 汇总报告
```

每个阶段的模式都是：**先跑已有 → 发现缺失 → 引导新建**。

## 详细流程

### 1. 选择需求

- 指定编号 → 使用该需求
- 未指定 → 查找可测试需求（状态为开发中/测试中）

### 2. 前置检查

- 状态必须为「开发中」或「测试中」
- 检查功能点完成情况，有未完成时警告

### 3. 提取测试要点

从需求文档提取测试要点，作为整个测试过程的验证基准：

```
📋 需求测试：REQ-001 <需求标题>

🧪 测试要点（共 N 项）：

API 测试：
- [ ] <API 测试点 1>
- [ ] <API 测试点 2>

业务规则测试：
- [ ] <业务规则测试点 1>
- [ ] <业务规则测试点 2>

数据权限测试：
- [ ] <数据权限测试点 1>

其他测试：
- [ ] <其他测试点>
```

### 4. 识别变更范围

从需求文档提取「文件改动清单」，定位本次需求涉及的源文件：

```
📁 分析变更范围...

从需求文档的「文件改动清单」提取：
├── <source-file-1>
├── <source-file-2>
├── <source-file-3>
└── <source-file-N>

关联的测试文件：
├── <test-file-1>               ✅ 存在
├── <test-file-2>               ✅ 存在
├── <test-file-3>               ❌ 不存在
└── <test-file-N>               ✅ 存在
```

**变更范围识别规则**：

1. **优先从需求文档**获取「文件改动清单」中列出的文件
2. 如果文档中没有明确列出，则通过 `git diff` 对比当前分支与主分支的变更文件
3. 按 CLAUDE.md 测试规范定位关联测试文件

## 测试模式说明

### 综合测试（默认）
按顺序执行 UT → API → E2E → 交互验证，某阶段失败时停止（除非 `--force`）。

### 增量测试
仅运行上次失败的用例：`/req:test REQ-XXX --failed`

### 跳过某阶段
```bash
/req:test REQ-001 --skip-e2e         # 跳过 E2E
/req:test REQ-001 --skip-ut          # 跳过 UT
/req:test REQ-001 --skip-api         # 跳过 API
```

### 强制继续
某阶段失败时继续后续阶段：`/req:test REQ-XXX --force`（不推荐）
