# 需求拆分分析

纯分析命令，帮助用户在创建需求前判断粒度是否合适，给出拆分或合并建议。

> 粒度规则参考 [_common.md](./_common.md)

## 命令格式

```
/req:split [需求描述或标题]
```

**示例：**
- `/req:split 用户积分系统`
- `/req:split 给订单列表加一个导出按钮`
- `/req:split 重构用户认证模块，支持 OAuth2.0 和 LDAP`

---

## 执行流程

### 1. 接收输入

- 用户提供需求标题或简要描述
- 无输入时提示用户描述想做的事情

### 2. 读取上下文

读取已有模块文档和需求列表，了解项目现状：

- `docs/requirements/modules/` — 现有模块及其职责
- `docs/requirements/active/` — 进行中的需求（避免重复）
- `docs/requirements/completed/` — 已完成的需求（了解历史）

### 3. 粒度分析

按以下维度逐项评估：

#### 3.1 粒度信号检测

| 信号 | 判断 | 建议 |
|------|------|------|
| 标题含「系统」「模块」「平台」「管理」 | 可能太大 | 列出子功能，建议拆分 |
| 标题含「接口」「字段」「按钮」「样式」 | 可能太小 | 合并到功能级 REQ 或改用 QUICK |
| 描述中包含多个独立的业务功能 | 太大 | 按可独立交付的功能拆分 |
| CRUD 属于同一业务实体 | 合适 | 增删改查放一个 REQ |
| 仅修改少量文件（< 5 个） | 可能太小 | 考虑用 QUICK |

**核心判断标准**：这个需求完成后，用户能感知到一个完整的功能变化吗？

#### 3.2 功能独立性分析

识别描述中包含的功能点，判断是否可独立交付：

- 功能可独立上线、独立使用 → 拆分
- 功能之间无强时序依赖（可并行开发） → 拆分
- 功能之间强耦合，必须同时上线 → 不拆
- 拆开后单个 REQ 无法独立验证 → 不拆

#### 3.3 前后端拆分判断

- 仅涉及后端 → 类型=后端，一个 REQ
- 仅涉及前端 → 类型=前端，一个 REQ
- 前后端都涉及，功能简单 → 类型=全栈，一个 REQ
- 前后端都涉及，功能复杂 → 拆分为后端 REQ + 前端 REQ，互相关联

#### 3.4 REQ vs QUICK 判断

| 场景 | 建议 |
|------|------|
| 新业务功能（CRUD、新页面） | REQ |
| 已有功能的小调整（加字段、改逻辑） | QUICK |
| Bug 修复 | QUICK |
| 重构/优化，改动 ≤ 5 个文件 | QUICK |
| 重构/优化，改动 > 5 个文件 | REQ |

### 4. 输出建议

根据分析结果，输出三种结论之一：

#### 结论一：粒度合适

```
📊 需求拆分分析

标题：用户积分规则管理-后端

📏 粒度判断：✅ 合适

分析：
- 功能完整：包含积分规则的 CRUD 和校验逻辑
- 可独立交付：完成后用户可以管理积分规则
- 改动范围适中：预估涉及 8-10 个文件

💡 下一步：
- /req:new 用户积分规则管理-后端 --type=后端
```

#### 结论二：建议拆分

```
📊 需求拆分分析

标题：用户积分系统

📏 粒度判断：⚠️ 太大，建议拆分

分析：
- 包含多个独立功能：积分规则管理、积分查询、积分兑换、积分排行
- 各功能可独立交付和验证
- 前后端都涉及，建议分开管理

📋 建议拆分为：
1. REQ: 用户积分规则管理-后端（类型=后端）
   - 积分规则 CRUD、规则校验
2. REQ: 用户积分规则管理-前端（类型=前端，关联=1）
   - 积分规则管理页面
3. REQ: 积分查询与展示-后端（类型=后端）
   - 积分余额查询、积分明细
4. REQ: 积分兑换-后端（类型=后端）
   - 兑换逻辑、库存扣减

💡 建议先创建：
- /req:new 用户积分规则管理-后端 --type=后端
```

#### 结论三：建议用 QUICK

```
📊 需求拆分分析

标题：给订单列表加一个导出按钮

📏 粒度判断：⚠️ 太小，建议用快速修复

分析：
- 改动范围小：仅涉及 1-2 个文件
- 风险低：不影响现有功能
- 无需完整设计和评审

💡 下一步：
- /req:new-quick 订单列表导出按钮
```

---

## 注意事项

- 此命令**只做分析，不创建文件**
- 分析基于用户描述和项目上下文，建议仅供参考
- 用户可以不采纳建议，直接按自己的判断创建需求
