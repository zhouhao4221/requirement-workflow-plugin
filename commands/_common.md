# 公共逻辑参考

> 此文档定义所有命令共用的逻辑，各命令直接引用，避免重复。

## 存储路径解析

```
本地存储（主）: docs/requirements/
├── modules/      # 模块文档
├── active/       # 进行中需求
├── completed/    # 已完成需求
└── INDEX.md      # 索引

全局缓存（副）: ~/.claude-requirements/projects/<project>/
```

**解析规则**：
1. 读取 `.claude/settings.local.json` 的 `requirementProject`
2. 有值 → 使用全局缓存路径
3. 无值 → 使用本地 `docs/requirements/`

**读取策略**：优先本地 → 本地不存在时从缓存读取

## 缓存同步规则

**触发时机**：任何需求文档的创建、修改、移动都必须同步缓存

| 操作 | 本地 | 缓存同步 |
|-----|------|---------|
| 创建需求 | 写入 active/ | 复制到缓存 active/ |
| 编辑需求 | 更新文件 | 复制到缓存覆盖 |
| 状态变更 | 更新文件 | 复制到缓存覆盖 |
| 完成归档 | 移动到 completed/ | 缓存同步移动 |

**同步逻辑**：
```bash
# 已绑定项目时执行同步
if [ -n "$PROJECT" ]; then
    cp $LOCAL_FILE $CACHE_PATH/
fi
```

**重要原则**：
1. **本地优先**：所有修改需求的命令（new、edit、review、dev、test、done）都必须先更新本地 `docs/requirements/` 中的文档
2. **本地成功后同步**：本地操作成功后立即同步到全局缓存
3. **本地不存在则跳过**：如果本地没有需求文档（如关联仓库），则不执行状态更新操作，避免数据不一致

## 状态更新确认机制

不同命令对状态更新的确认要求：

| 命令 | 状态变更 | 确认机制 |
|-----|---------|---------|
| `/req:review pass/reject` | 待评审 → 评审通过/驳回 | 显式参数即为确认 |
| `/req:dev` | 评审通过 → 开发中 | 首次进入时自动更新 |
| `/req:test` | 开发中 → 测试中 | 执行命令即为确认 |
| `/req:done` | 测试中 → 已完成 | **必须明确确认（y/n）** |

## 需求编号生成

扫描 active/ 和 completed/ 目录，找最大编号 +1，格式 `REQ-XXX`

## 状态流转

```
📝 草稿 → 👀 待评审 → ✅ 评审通过 → 🔨 开发中 → 🧪 测试中 → 🎉 已完成
```

## 元信息字段

| 字段 | 说明 |
|------|------|
| 编号 | REQ-XXX |
| 类型 | 后端/前端/全栈 |
| 状态 | 当前状态 |
| 模块 | 所属模块 |
| 关联需求 | 前后端对应需求 |
