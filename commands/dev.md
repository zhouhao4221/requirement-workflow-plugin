# 需求开发

启动或继续需求开发，生成开发任务并逐步实现。

> 存储路径规则见 [_common.md](./_common.md)

## 命令格式

```
/req:dev [REQ-XXX] [--reset]
```

- 省略编号时自动选择「评审通过」或「开发中」的需求
- `--reset` 强制从头开始

---

## 执行流程

### 1. 选择需求

- 指定编号 → 使用该需求
- 未指定 → 查找可开发需求（状态为评审通过/开发中）
- 多个候选 → 让用户选择

### 2. 前置检查（严格执行）

**根据需求类型区分检查规则**：

#### 正式需求 (REQ-XXX) - 必须通过评审

| 当前状态 | 处理方式 |
|---------|---------|
| 📝 草稿 | ❌ 拒绝开发，提示：`需求尚未评审，请先执行 /req:review 提交评审` |
| 👀 待评审 | ❌ 拒绝开发，提示：`需求正在评审中，请等待评审通过后再开发` |
| ✅ 评审通过 | ✅ 允许开发，首次进入自动更新状态为「开发中」 |
| 🔨 开发中 | ✅ 允许继续开发 |
| 🧪 测试中 | ✅ 允许开发（可能是修复测试问题） |
| 🎉 已完成 | ⚠️ 警告提示：`需求已完成，如需修改请创建新需求` |

#### 快速修复 (QUICK-XXX) - 跳过评审

| 当前状态 | 处理方式 |
|---------|---------|
| 📝 草稿 | ✅ 允许开发（方案确认后直接开发） |
| ✅ 方案确认 | ✅ 允许开发，自动更新状态为「开发中」 |
| 🔨 开发中 | ✅ 允许继续开发 |
| 🎉 已完成 | ⚠️ 警告提示：`需求已完成，如需修改请创建新需求` |

**重要**：正式需求 (REQ) 未通过评审**不能**开始开发；快速修复 (QUICK) 跳过评审环节。

### 3. 加载上下文

读取需求文档：功能清单、文件改动清单、实现步骤、数据模型、API 设计

### 4. 显示开发概览

```
📋 REQ-001 部门渠道关联

📊 进度：2/6 功能点已完成
📝 功能清单：
- [x] Model/Store 层
- [x] 渠道范围校验
- [ ] 获取可选渠道接口 ← 当前
- [ ] 订单数据过滤
...
```

### 5. 更新状态

首次进入 → 状态改为「开发中」

### 6. 生成任务

根据实现步骤生成 TodoWrite 任务列表

### 7. 逐步实现

按分层架构顺序开发：Model → Store → Biz → Controller → Router

实时检查：
- 文件命名（kebab-case）
- 日志规范、错误处理
- 多租户、Swagger 注解

### 8. 进度更新

每完成一步：更新任务状态 + 需求文档 checkbox

### 9. 开发完成

```
🎉 开发完成！
- 功能点：6/6 ✅
- 新增/修改文件统计

💡 下一步：
- /req:test REQ-001 - 进入测试
```
