# v2.12.0

> 发布日期：2026-04-18
> 版本范围：v2.11.0..HEAD

## 新功能

- **`/req:release` 新增 `release-branch` 流程模式**(req 3.8.0):当命令在 `release/*` 或 `chore/release-*` 分支上执行时,自动走标准 git-flow 双 PR 流程——
  - **PR1: release 分支 → main**,先合并让 tag 落在干净的 main HEAD 上(只含本次发布内容,不被 develop 上未准备发布的 feature 污染)
  - **PR2: release 分支 → develop**,后合并回流 release 准备 commit(changelog + SQL + 回滚脚本),避免下次 release 重复生成
  - **PR 顺序不可颠倒**:PR1 必须在 PR2 前合并,否则 tag 会被 develop 上的 feature 污染
  - **PR2 为非阻塞交互**:tag 和 Release 完成后,用户可选"跳过"PR2 等待直接进入报告,PR2 保留等后续手动合并

## 行为变化

- **Gitea 非 draft 场景不再本地推 tag**:由 Gitea Release API 从 `target_commitish=tag_target` 在主分支 HEAD 自动生成 lightweight tag,和用户实际工作流一致(tag 在创建 Release 时一并出现)。新的 `PUSH_TAG_FIRST` 决策矩阵:
  - 非 draft + gitea → **false**(新行为,原先是 true)
  - 非 draft + github / other → true(保持)
  - draft + gitea → true(保持,API 要求 tag 先存在)
  - draft + github → false(保持,lazy 创建)
- **git-flow 保护主分支守门提示扩展**:`/req:release` 在 git-flow + 保护主分支上起步时,除了推荐切到 develop 走 cross-branch 流程外,新增"切到/新建 release 分支走 release-branch 流程"选项,指导用户隔离 develop 上未准备发布的 feature。
- **`--no-draft` 二次确认扩展到 release 分支**:防止用户在 release 分支上误用 `--no-draft` 跳过 draft 闸门。
- **`/req:changelog` 固定使用 haiku 模型**:命令规则明确(按 Git 记录分类 commit),显式降级到 haiku 降低调用成本。

## 边界情况更新

- 新增 `release-branch` 流程 PR1 中止与 PR2 跳过处理条目
- 更新 Gitea `Release is has no Tag`(422)错误说明:仅在 **draft + gitea** 场景触发,非 draft + gitea 由 API 自行生成 tag 不会出错
- 更新"`--no-draft` 在受保护主分支 + 跨分支流程"场景:按 `repoType` 分叉展开(github 推荐回默认 draft 绕开 tag 保护;gitea 回默认 draft 反而会踩 tag push,需先解除 tag 保护)

## 文档

- 起步分支速查表加入 cross-branch vs release-branch 的选择指南:develop 到 main 的 delta 就是要发的全部内容 → cross-branch;develop 上还有未就绪 feature → release-branch。
