# 需求分析：方法论对照与设计评审

> 本文记录 DevFlow `req` 插件中需求分析流程（`/req:new`、`/req:edit`、`requirement-analyzer` 技能、`requirement-template.md`）相对于经典需求分析方法论的符合度评估与改进建议，用于后续讨论和迭代。
>
> 最近一次评估：2026-04-17

---

## 参考方法论

本文对照以下主流需求分析方法论进行评估：

- **IEEE 830 / ISO/IEC/IEEE 29148** —— SRS（软件需求规格说明）标准
- **Volere Requirements Template** —— Robertson & Robertson 的综合模板
- **BABOK** —— IIBA 业务分析知识体系
- **RUP Use Case** —— 角色、前置条件、基本流程、备选流程
- **FURPS+** —— Functional / Usability / Reliability / Performance / Supportability
- **ISO 25010** —— 软件质量模型（NFR 分类）
- **Agile / User Story** —— INVEST 原则、Given-When-Then 验收标准
- **Kano Model** —— 必备 / 期望 / 魅力 / 无差异 / 反向需求分类
- **MoSCoW** —— Must / Should / Could / Won't 优先级
- **SMART** —— 目标具体、可衡量、可达成、相关、有时限
- **5W2H / 5 Why** —— 完整性与根因挖掘

---

## 当前设计相符之处

| 维度 | 对应方法论 | 实现位置 |
|------|----------|---------|
| 探索 → 确认 → 生成 两阶段 | Agile Discovery / Design Thinking | `SKILL.md` 阶段一/阶段二 |
| 4 轮提问（痛点 / 期望 / 流程 / 约束） | 5W2H + FURPS+ | `SKILL.md` 提问流程 |
| 使用场景「角色 / 前置 / 基本流 / 异常」 | RUP Use Case | 模板第四章 |
| 业务规则独立成章 | BABOK Business Rules | 模板第三章 |
| 多轮不限、用户确认后再落盘 | 用户主导 / 避免过早具体化 | `SKILL.md` 等待确认 |
| 粒度 / 功能扩展判断 | INVEST-Small + 变更控制 | `SKILL.md` 粒度检查 |
| 编辑时意图澄清 + 关联影响提示 | Impact Analysis | `SKILL.md` 编辑原则 |
| 量化效果目标 | SMART-Measurable | 第 2 轮追问 |
| 验收标准分「技术测试 + 产品验收」 | V-Model / Acceptance Criteria | 模板 6.1 / 6.2 |
| 需求类型（后端 / 前端 / 全栈）分析侧重点不同 | 上下文适配 | `SKILL.md` 识别需求类型 |
| 模板章节结构不可变、占位保留 | 文档一致性 / 可追溯性 | `SKILL.md` 格式约束 |

### 亮点

- **两阶段流程**（先多轮讨论收集，用户确认后一次性生成）有效避免 AI 提前推演填充而用户无感的 anti-pattern。
- **「去掉这个功能点原需求能否独立交付」** 作为拆分判断核心问题，对应 INVEST-Independent 原则，简洁有效。
- **「以本地为准 + 强制自动同步」** 的缓存策略让需求文档具备可审查性（纳入 Git）又具备跨仓库共享能力，符合"需求是团队共享资产"的基本立场。

## 方法论缺口与改进建议

### 1. 干系人（Stakeholders）识别未显式化

**问题**：当前只问「谁反馈的」「角色」，未覆盖业务方、合规、运维、客服等非最终用户干系人。

**方法论依据**：Volere / BABOK 均要求在需求分析起点明确 **Stakeholder Map**。

**建议**：
- `SKILL.md` 第 1 轮追问增加「除了提出方，还有谁会因此变化受影响？（运营 / 客服 / 财务 / 合规 / 运维）」
- 模板第一章增加 **1.5 干系人** 小节

---

### 2. 缺少范围边界（In / Out of Scope）

**问题**：第一章只有背景 / 目标 / 客户场景 / 价值，没有明确声明「不做什么」。

**方法论依据**：IEEE 830 的「Product Scope」、Volere 的「Scope of the Work」都强调显式 Out-of-Scope。这是防止 scope creep 的第一道防线。

**建议**：模板第一章增加 **1.x 范围与边界**：
```markdown
### 1.x 范围与边界

- **本期包含**：...
- **本期不做**：明确声明哪些相关但不纳入当前需求的场景
```

---

### 3. 缺少假设（Assumptions）

**问题**：第十章关联信息里只有「外部依赖」和「风险」，未覆盖「假设」。

**方法论依据**：Volere 把 Assumptions 和 Constraints、Dependencies 并列为三种项目约束。假设一旦不成立就是风险，但性质不同——假设是"我们相信 X 是真的"，依赖是"我们必须拿到 Y"。

**建议**：第十章增加 `**假设**：...` 字段，例如「假设用户积分规则变更频率低于 1 次/月」「假设第三方短信服务可用率 > 99.9%」。

---

### 4. NFR（非功能需求）未独立分类

**问题**：NFR 当前塞在业务规则表的最后一行「非功能约束」，以及第 4 轮提问的综合一问。

**方法论依据**：FURPS+ / ISO 25010 建议 NFR 按多维度分类：
- **性能**（响应时间、吞吐量、并发量、数据量级）
- **可用性**（SLA、降级策略、灾备）
- **安全**（鉴权、脱敏、审计、越权防护）
- **可维护性**（监控、日志、可观测性）
- **兼容性**（浏览器、设备、老接口、多端）

**建议**：
- 模板第三章将「非功能约束」从一行拆成独立的 3.x 子章节（或独立为第三章 B 组）
- `SKILL.md` 第 4 轮提问按这 5 个维度逐项追问（或至少给出 checklist）

---

### 5. 数据需求的概念建模缺位

**问题**：数据模型（11.1）完全推迟到 `/req:dev` 阶段。

**风险**：需求阶段不讨论核心业务实体 / 关键属性 / 关系，会导致：
- 功能清单脱离数据语义（"查询订单"但不清楚订单实体结构）
- 业务规则难以表达（"只有管理员能改订单状态"——订单有什么状态？）

**方法论依据**：BABOK 将 Data Modeling 视为需求分析的核心技术之一，分为**概念模型**（业务语义）和**逻辑/物理模型**（技术实现）。前者属于需求阶段。

**建议**：模板第五章加入 **5.x 业务实体与数据需求**（仅概念级）：
```markdown
### 5.x 业务实体

| 实体 | 关键属性（业务概念） | 与其他实体关系 |
|------|--------------------|---------------|
| 订单 | 编号、状态、金额、客户、下单时间 | 1-N 订单项 |
| 订单项 | 商品、数量、单价 | N-1 订单 |
```

注意：**只描述业务语义，不定义字段类型 / 表结构 / 主外键**，后者仍在 `/req:dev` 生成。

---

### 6. 验收标准未采用 Given-When-Then 格式

**问题**：当前 6.2 是「在什么页面 → 执行什么 → 能看到什么」，接近但不是标准 AC 格式。

**方法论依据**：Agile / BDD 社区推崇 Given-When-Then 三段式 AC，可测性更强，自动化测试用例可直接映射。

**建议**（非强制，可选）：
```markdown
### 6.2 验收标准（Given-When-Then）

- **AC-1**：
  - **Given** 用户已登录且积分余额 ≥ 100
  - **When** 点击「兑换」按钮
  - **Then** 扣减 100 积分 + 订单状态流转为「待发货」
```

两种格式可共存，让用户选择。

---

### 7. 功能清单缺少功能级优先级

**问题**：优先级只在元信息（需求级 P1/P2/P3），功能点级未标注。

**方法论依据**：MoSCoW（Must / Should / Could / Won't）常用于功能点级别排序，避免"全部必做"陷阱，便于分期交付。

**建议**：模板第二章功能清单加优先级列：
```markdown
- [ ] **[M] 功能点1**：核心必做
- [ ] **[S] 功能点2**：应做
- [ ] **[C] 功能点3**：可做
- [ ] **[W] 功能点4**：本期不做（留档）
```

---

### 8. 粒度判断缺少估算维度

**问题**：当前粒度判断只看「能否独立交付」（对应 INVEST-Independent 和 Small），未覆盖 Estimable / Negotiable。

**方法论依据**：INVEST 六原则缺一不可。Estimable 要求"能估出工作量"，过大/过模糊的需求无法估算。

**建议**：`SKILL.md` 粒度检查增加一条：
| 信号 | 判断 |
|------|------|
| 能估出 1-5 天开发量 | 合适 |
| 估不准、涉及未知技术探索 | 太大，先拆一个 spike |
| 不足 0.5 天 | 太小，合并到相关功能或用 `/req:new-quick` |

---

### 9. 缺少 5 Why 深挖

**问题**：`SKILL.md` 的追问只在「回答模糊时针对性追问」，未强制做因果链挖掘。

**风险**：容易停在表层症状，错过真正要解决的业务问题。例如用户说"想要一个导出按钮"，不追问"为什么要导出 → 要发给财务 → 财务为什么不直接看 → 因为财务系统没对接"——真正的需求可能是系统对接而非导出按钮。

**建议**：`SKILL.md` 第 1 轮追问加入 5 Why 触发条件：「用户描述的是**解决方案**而非**问题**时，追问『为什么需要这个？如果没有会怎样？』」

---

### 10. 效果目标 → 度量闭环未强制

**问题**：模板 6.2 只说「如有量化目标，说明如何确认达成」，但没要求回到第 1.2 节的「效果目标」做闭环。

**风险**：定了 "提效 30%" 但没定义怎么测——上线后无法验证是否达成。

**建议**：
- `SKILL.md` 第 2 轮追问 `"能举个例子说明好用具体是什么样吗？"` 后，追加 `"上线后靠什么数据判断达成了？（埋点 / 报表 / SLA 监控）"`
- 模板 1.2 效果目标旁边加「**度量方式**：...」子字段

---

### 11. 集成需求无独立位置

**问题**：跨系统集成（第三方 API / 消息队列 / 定时任务 / Webhook）混在第十章「外部依赖」里。

**方法论依据**：SRS 标准把「External Interface Requirements」作为独立顶层章节（用户接口、硬件接口、软件接口、通信接口）。

**建议**：模板第五章除了「接口需求」「交互逻辑」外，加一节 **5.x 集成需求**：
```markdown
### 5.x 集成需求

| 集成类型 | 对端 | 方向 | 触发时机 | 业务语义 |
|---------|------|------|---------|---------|
| REST API | 短信服务商 | 调出 | 订单创建后 | 发送订单确认短信 |
| MQ | 订单中心 | 订阅 | 订单状态变更 | 同步到积分系统 |
| Cron | - | - | 每日 02:00 | 清算昨日积分 |
```

## 小步迭代优先级

按改进收益 / 改动成本比，建议分三期落地：

### 第一期（高收益，低改动）✅ 已落地（2026-04-17）

1. 模板第一章加 **1.5 范围边界（In/Out of Scope）** + **1.6 干系人**
2. 模板第十章加 **假设（Assumptions）** 字段
3. `SKILL.md` 第 1 轮追问加 5 Why 触发条件

**落地记录**：

| 文件 | 改动内容 |
|------|---------|
| `plugins/req/templates/requirement-template.md` | 第一章新增 `1.5 范围与边界`（本期包含/不做两列）、`1.6 干系人`（角色/关注点/备注三列表格） |
| `plugins/req/templates/requirement-template.md` | 第十章关联信息新增 **假设** 字段，附"假设 vs 依赖"区分说明 |
| `plugins/req/skills/requirement-analyzer/SKILL.md` | 第 1 轮提问：质量门槛增加「识别至少一类干系人」；追问触发新增三条：5 Why 深挖（解决方案 → 真问题）、干系人识别、范围边界 |
| `plugins/req/skills/requirement-analyzer/SKILL.md` | 阶段二生成：补上 1.5、1.6 填充规则；第十章加"假设"提取说明（区分 Assumption vs Dependency） |
| `plugins/req/commands/new.md` | 阶段二概述同步加入范围边界、干系人、假设，保持与 SKILL.md 一致 |

**兼容性处理**：已有需求文档不会被动改动；`/req:edit` 时由用户决定是否补齐新章节；`/req:update-template` 可一键同步新模板到项目本地，不覆盖需求文档。

### 第二期（方法论补齐）

4. 模板第三章「非功能约束」拆成独立的 3.x 子章节（性能 / 安全 / 兼容 / 可用 / 可维护）
5. 模板第五章加 **5.x 业务实体与数据需求**（概念级）
6. 模板第二章功能清单加 MoSCoW 优先级列
7. `SKILL.md` 第 2 轮追问加「效果目标如何度量」

### 第三期（可选增强）

8. 6.2 验收标准支持 Given-When-Then 格式（与现有格式并存）
9. 模板第五章加 **5.x 集成需求** 表
10. `SKILL.md` 粒度检查加入估算维度

## 兼容性与迁移策略

修改模板会影响已有需求文档。建议策略：

- **新需求**：直接使用新模板
- **已有需求**：保持原结构，不强制迁移；编辑时由用户决定是否补齐新增章节
- **版本标记**：模板顶部加 `version: 2` 字段，`requirement-analyzer` 读取时做兼容分支
- **`/req:update-template` 命令**：提供一键同步最新模板到项目本地，但不覆盖已有需求文档

## 遗留讨论点

以下问题尚未决定，留待后续讨论：

1. **是否强制 Stakeholder Map？** 小团队 / 内部工具可能过度，是否仅作为可选字段？
2. **Kano Model 分类是否引入？** 对"必备"和"魅力"功能做差异化设计，但会增加提问成本。
3. **用户故事（User Story）格式是否作为可选输出？** 当前使用场景 + 功能清单组合可读性强，是否也支持 `As a / I want / So that` 格式导出以对接 Jira / 禅道？
4. **需求间的依赖图是否需要显式化？** 当前只在「关联需求」里列编号，没有 DAG 视图；是否需要 `/req:graph` 命令？
